# 为什么我把 AI Agent 的上下文扔了 89.6%，它反而连续跑了 72 小时不崩溃？

> 一个反直觉的发现：上下文管理的本质不是「记住更多」，而是「不再重复走死路」。

---

## 一、 turn 847 的崩溃

我跑一个 OpenClaw agent 做深度代码分析和研究任务。几个月来，我反复目睹同一个死亡螺旋：

- **第 200 轮**：开始出现轻微漂移。
- **第 500 轮**：agent 提出了一个我明明已经否定过两次的错误修复方案。
- **第 847 轮**：`contextUsage` 飙到 **111.9%**，gateway 触发紧急压缩，agent 在 3 分钟内被塞了 6 次截断的上下文。
- **第 900 轮**：我不得不手动重启会话，因为 agent 已经彻底忘记了任务的核心约束。

行业里所有人都在说同一个答案：**买更大的上下文窗口。**

我试过 128K，试过 1M，甚至试过 2M。崩溃照旧发生。

因为问题从来不是「记不住」，而是**「记错了方向」**。

---

## 二、所有现有方案都在回答一个错误的问题

RAG 在问：「什么和 query 语义最相似？」
MemGPT 在问：「模型自己觉得该记住什么？」
Gemini 在问：「2M token 够不够放？」

但它们都漏了真正致命的问题：

> **「LLM 在推进当前论点的方向上，有没有在重复论证同一个已知的错误？」**

这不是正向保证（必能找到正确路径），而是**负向保证**（不重复走死路）。

当你把 agent 的上下文从 100% 硬塞到 10.39%，剩下的不是随机丢弃，而是**只保留与当前语义方向对齐的证据**。agent 不再被海量无关历史淹没，也不会在错误方向上反复打转。

---

## 三、CDA 是什么？

**CDA（Context Direction Alignment，上下文方向对齐）** 不是一个更大的内存仓库，而是一把**指南针**。

它做了三件反直觉的事：

### 1. 方向对齐 > 容量扩张
传统方案把上下文当成等权重的序列。CDA 把它当成**语义梯度场**——在证据-论点耦合的方向上密度大，在正交方向上密度趋近于零。

### 2. 负向过滤 > 正向搜索
CDA 不保证第一次就找对路。但它保证：**同一个死路不会被走第二次**。

失败的方向会被记录为 `miss`，在下一次 `assemble` 时权重衰减；连续两次 `miss` 后，该方向直接被跳过。

### 3. 拓扑压缩 > 截断摘要
SCG（Semantic Compression Graph）压缩时保留的不是 token，而是**语义拓扑**。即使 90% 的 token 消失，片段之间的逻辑关系仍然完整。

## 五、Phase 驱动的五阶段框架

CDA 把 agent 的生命周期切成了五个相位，每个相位拥有独立的上下文策略：

- **`assemble`**：构建当前 turn 的输入，过滤已知死路
- **`ingest`**：分块并标记新消息的语义方向
- **`afterTurn`**：评估对齐分数，记录 `miss`
- **`compact`**：用 SCG 保留拓扑结构地压缩
- **`bootstrap`**：冷启动时优先加载热经验

上下文不再被时间或 token 数量驱动，而是被**语义状态转移**驱动。
